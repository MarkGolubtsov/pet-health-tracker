# Технологический стек Pet Health Tracker

Документ фиксирует технологический стек первой версии продукта.
Стек выбран под быстрый выпуск, хранение медицинских документов,
серверную авторизацию и последующее расширение без смены платформы.

## Архитектурное решение

1. Frontend реализуется как отдельный web-модуль.
2. Backend реализуется как отдельный серверный интерфейс.
3. Supabase используется как основная платформа данных.
4. Supabase Auth используется как механизм авторизации.
5. Supabase Storage используется как файловое хранилище.
6. OpenAI Vision используется для распознавания лабораторных документов.
7. Backend остается владельцем служебных операций и интеграций.

## Frontend

1. Стек frontend-модуля имеет статус `TBD`.
2. Frontend-стек должен быть выбран отдельным архитектурным решением.
3. До выбора стека frontend-модуль остается контейнерным placeholder.

## Backend

1. `Node.js 22` используется как среда выполнения backend-модуля.
2. `TypeScript` используется как обязательный язык backend-модуля.
3. `Fastify` используется как серверный фреймворк.
4. `@fastify/multipart` используется для приема файлов, если загрузка идет
   через backend.
5. `Zod` используется для схем доменных данных и результата распознавания.
6. `Playwright` используется для генерации отчетов в формате PDF.
7. Backend предоставляет публичный серверный интерфейс для сценариев продукта.
8. Backend не хранит файлы на локальной файловой системе контейнера.

## Платформа данных

1. `Supabase Postgres` используется как основная база данных.
2. `Supabase Auth` используется для пользователей и сессий.
3. `Supabase Storage` используется для исходных документов анализов.
4. `Supabase Row Level Security` используется для доступа на уровне строк.
5. `Supabase migrations` используются для схемы базы данных.
6. `Supabase migrations` используются для политик файлового хранилища.

## Авторизация

1. Первая версия использует регистрацию по email и паролю.
2. Первая версия использует вход по email и паролю.
3. Пользователь входит с разных устройств через один аккаунт.
4. Email должен быть подтвержден до работы с медицинскими данными.
5. Сохранение профиля, документов и анализов требует постоянного аккаунта.
6. Данные пользователя связываются с `auth.uid()`.
7. Таблицы продукта должны иметь `owner_id`.
8. Политика доступа должна разрешать чтение только владельцу записи.
9. Восстановление пароля входит в контракт авторизации первой версии.
10. Анонимный вход не используется для медицинских данных.
11. Вход через внешний провайдер добавляется как расширение.
12. Служебный ключ Supabase используется только на backend.
13. Служебный ключ Supabase используется только для служебных операций.

## Файловое хранилище

1. Исходные документы анализов хранятся в `lab-documents`.
2. `lab-documents` является приватным файловым контейнером.
3. Файловый контейнер принимает изображения и файлы в формате PDF.
4. Файловый контейнер принимает сканы лабораторных бланков.
5. Объект хранилища связывается с записью анализа в Postgres.
6. Путь объекта должен включать `owner_id`.
7. Доступ к документам идет через политику доступа или подписанную ссылку.
8. Публичные файловые контейнеры не используются для медицинских документов.
9. Локальная файловая система контейнеров используется только временно.
10. Временное хранение допускается только во время обработки запроса.

## Распознавание документов

1. OpenAI Vision используется для извлечения данных из фото и сканов.
2. Файл в формате PDF передается в процесс распознавания как документ.
3. Результат распознавания возвращается в структурированном формате.
4. Backend валидирует результат распознавания через доменную схему.
5. Неопознанные значения сохраняются для экрана проверки.
6. Сомнительные значения сохраняются для экрана проверки.
7. Распознавание не создает финальный анализ без подтверждения пользователя.

## Основные зависимости первой версии

1. `@supabase/supabase-js` используется в backend.
2. `openai` используется в backend для распознавания документов.
3. `fastify` используется в backend для серверного интерфейса.
4. `zod` используется для схем входных и выходных данных.
5. `playwright` используется в backend для экспорта отчетов в PDF.
6. Зависимости frontend-модуля имеют статус `TBD`.

## Границы ответственности

1. Frontend отвечает за пользовательские сценарии.
2. Frontend отвечает за экран проверки распознанных данных.
3. Frontend не содержит серверные бизнес-правила.
4. Backend отвечает за распознавание документов.
5. Backend отвечает за экспорт отчетов в формате PDF.
6. Backend отвечает за служебные операции.
7. Backend не реализует собственное файловое хранилище.
8. Supabase отвечает за хранение данных.
9. Supabase отвечает за авторизацию.
10. Supabase отвечает за файловое хранилище.
11. Supabase отвечает за контроль доступа.
12. Политики доступа являются обязательной частью модели безопасности.

## Не входит в первую версию

1. Собственный сервис авторизации.
2. Собственное объектное хранилище.
3. Kubernetes.
4. Шина событий.
5. Отдельная очередь задач.
6. Мобильное приложение.
7. Диагностические медицинские рекомендации.

## Локальная разработка

1. Локальный запуск frontend зависит от выбранного frontend-стека.
2. Backend запускается как сервис Node.js.
3. Supabase для локальной разработки запускается через командную строку.
4. Supabase может использоваться как управляемый проект для разработки.
5. Docker Compose остается основным способом поднять контур проекта.
6. Переменные окружения фиксируются в `.env.example`.

## Решения к пересмотру после первой версии

1. Перенести распознавание в очередь, если обработка начнет блокировать сервер.
2. Добавить вход через внешний провайдер.
3. Добавить фоновые процессы для массовой обработки документов.
4. Добавить другого провайдера хранилища при требовании к переносимости.
5. Вынести контракт серверного интерфейса, если появятся внешние клиенты.
