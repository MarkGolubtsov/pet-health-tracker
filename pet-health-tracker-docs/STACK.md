# Технологический стек Pet Health Tracker

Документ фиксирует технологический стек первой версии продукта.
Стек выбран под deploy через Docker Compose, единый `.env`, хранение
медицинских документов и быстрый выпуск первой версии.

## Архитектурное решение

1. Frontend реализуется как отдельный web-модуль.
2. Backend реализуется как отдельный серверный интерфейс.
3. Backend является единственным потребителем базы данных.
4. Backend является единственным потребителем файлового хранилища.
5. Backend владеет авторизацией пользователей.
6. Backend владеет миграциями базы данных.
7. Postgres используется как основная база данных.
8. MinIO используется как файловое хранилище первой версии.
9. Mailpit используется только для локальной проверки email-сценариев.
10. OpenAI Vision используется для распознавания лабораторных документов.

## Схема взаимодействия

1. Frontend обращается только к backend.
2. Backend обращается к Postgres.
3. Backend обращается к файловому хранилищу.
4. Backend обращается к SMTP-сервису.
5. Backend обращается к сервису распознавания документов.

## Frontend

1. Стек frontend-модуля имеет статус `TBD`.
2. Frontend-стек должен быть выбран отдельным архитектурным решением.
3. До выбора стека frontend-модуль остается контейнерным placeholder.

## Backend

1. `Node.js 22` используется как среда выполнения backend-модуля.
2. `TypeScript` используется как обязательный язык backend-модуля.
3. `Fastify` используется как серверный фреймворк.
4. `@fastify/multipart` используется для приема файлов.
5. `Zod` используется для схем входных и выходных данных.
6. `argon2` используется для хэширования паролей.
7. `pg` используется для подключения к Postgres.
8. `@aws-sdk/client-s3` используется для работы с файловым хранилищем.
9. `nodemailer` используется для отправки писем.
10. `Playwright` используется для генерации отчетов в формате PDF.
11. Backend предоставляет публичный серверный интерфейс для продукта.
12. Backend не хранит файлы на локальной файловой системе контейнера.

## База данных

1. Postgres хранит данные продукта.
2. Схема базы данных принадлежит backend-модулю.
3. Миграции находятся в `pet-health-tracker-backend/migrations`.
4. Seed-данные находятся в `pet-health-tracker-backend/seeds`.
5. Docker volume хранит локальные данные Postgres.
6. Production backup базы данных является отдельной операционной задачей.

## Авторизация

1. Первая версия использует регистрацию по email и паролю.
2. Первая версия использует вход по email и паролю.
3. Пользователь входит с разных устройств через один аккаунт.
4. Email должен быть подтвержден до работы с медицинскими данными.
5. Сохранение профиля, документов и анализов требует постоянного аккаунта.
6. Пароли хранятся только в виде стойкого хэша.
7. Backend выдает сессию после успешного входа.
8. Сессия должна храниться в защищенной cookie.
9. Восстановление пароля входит в контракт авторизации первой версии.
10. Анонимный вход не используется для медицинских данных.
11. Вход через внешний провайдер добавляется как расширение.

## Файловое хранилище

1. Исходные документы анализов хранятся в bucket `lab-documents`.
2. Bucket должен быть приватным.
3. Первая версия использует MinIO как реализацию файлового хранилища.
4. Backend загружает файлы в хранилище.
5. Backend проверяет права перед выдачей доступа к файлу.
6. Backend выдает подписанную ссылку только после проверки прав.
7. Путь объекта должен включать идентификатор владельца.
8. Публичный доступ к медицинским документам запрещен.
9. Локальная файловая система контейнеров используется только временно.
10. Временное хранение допускается только во время обработки запроса.
11. Production MinIO должен использовать persistent volume.
12. Production MinIO должен иметь backup-процесс.

## Email

1. Backend отправляет письма через SMTP.
2. Локальная разработка использует Mailpit через Compose profile `dev`.
3. Mailpit принимает письма подтверждения email.
4. Mailpit принимает письма восстановления пароля.
5. Production не должен использовать Mailpit.
6. Production SMTP-провайдер задается через переменные окружения.

## Распознавание документов

1. OpenAI Vision используется для извлечения данных из фото и сканов.
2. Файл в формате PDF передается в процесс распознавания как документ.
3. Результат распознавания возвращается в структурированном формате.
4. Backend валидирует результат распознавания через доменную схему.
5. Неопознанные значения сохраняются для экрана проверки.
6. Сомнительные значения сохраняются для экрана проверки.
7. Распознавание не создает финальный анализ без подтверждения пользователя.

## Docker Compose

1. Docker Compose запускает frontend.
2. Docker Compose запускает backend.
3. Docker Compose запускает Postgres.
4. Docker Compose запускает MinIO.
5. Docker Compose создает bucket `lab-documents`.
6. Docker Compose запускает Mailpit только через profile `dev`.
7. `.env` является источником локальных портов и секретов разработки.
8. Production deploy может использовать тот же compose-контракт.
9. Production deploy не включает profile `dev`.
10. Production deploy включает MinIO.
11. Infrastructure ports должны быть привязаны к loopback в production.

## Границы ответственности

1. Frontend отвечает за пользовательские сценарии.
2. Frontend отвечает за экран проверки распознанных данных.
3. Frontend не содержит серверные бизнес-правила.
4. Backend отвечает за авторизацию.
5. Backend отвечает за доступ к данным.
6. Backend отвечает за миграции базы данных.
7. Backend отвечает за доступ к файлам.
8. Backend отвечает за распознавание документов.
9. Backend отвечает за экспорт отчетов в формате PDF.
10. Postgres отвечает за хранение структурированных данных.
11. MinIO отвечает за хранение исходных документов.
12. Mailpit отвечает только за локальную проверку писем.
13. Production SMTP-провайдер отвечает за доставку писем пользователям.

## Не входит в первую версию

1. Отдельный сервис авторизации.
2. Managed backend-as-a-service платформа.
3. Self-hosted backend-as-a-service stack.
4. Kubernetes.
5. Шина событий.
6. Отдельная очередь задач.
7. Мобильное приложение.
8. Диагностические медицинские рекомендации.

## Локальная разработка

1. Локальный запуск frontend зависит от выбранного frontend-стека.
2. Backend запускается как сервис Node.js.
3. Postgres запускается через Docker Compose.
4. MinIO запускается через Docker Compose.
5. Mailpit запускается через Docker Compose profile `dev`.
6. Переменные окружения фиксируются в `.env.example`.

## Решения к пересмотру после первой версии

1. Перенести распознавание в очередь, если обработка начнет блокировать сервер.
2. Добавить вход через внешний провайдер.
3. Добавить фоновые процессы для массовой обработки документов.
4. Заменить MinIO на внешний S3-compatible storage при необходимости.
5. Вынести контракт серверного интерфейса, если появятся внешние клиенты.
